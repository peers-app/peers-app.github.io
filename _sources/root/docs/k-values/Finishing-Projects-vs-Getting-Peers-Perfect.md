I've repeatedly heard great arguments for why it is so important to prioritize "finishing projects" when working on personal projects.  I agree with them.  Iterating is easy. Producing that first thing to iterate on is hard.  If the v1 of the project is never delivered then the world never has an opportunity to judge it and decide if it's valuable.  

The issue with taking this approach with Peers is that Peers is my platform to deliver a huge number of personal projects that I've been stewing on for a long time.  Apps, games, etc.  Because it's basically a meta-project or precursor project, I think it's much more important to "get it right".  Additionally, if Peers is successful and *other people* start building on top of it then it could be *much* harder to go back and fix some severe structural problem.  

So spending more time ensuring Peers is a stable and well structured platform to build other projects on top of (anything from simple, one-table, CRUD apps all the way to full-blow canvas based, multiplayer games) seems like a worth-while endeavor and good enough arguments against the get-it-out-fast-at-almost-any-cost-and-with-warts argument.
