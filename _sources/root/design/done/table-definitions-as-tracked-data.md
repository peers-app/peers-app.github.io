
Currently syncing data can still be flaky and unreliable, and I believe it's almost entirely due to table definitions (schemas) not yet existing because packages have not been synced or are out of date. I think the easiest and most reliable way to fix this is to automatically track table definitions in a separate table, so we need to update the table class to insert the table definition into its own table when it's instantiated (this is a nice approach since it doesn't require code to be changed in many places and, I think, will fully solve the problem with the added benefit of other peers not having to have packages installed which defines tables to be able to instantiate and sync that table). 

There are a few important cases to think about:

The first is how do we deal with a table class instance when the table definition has been updated. I suspect that can be done fairly easily by just running the internal code to make sure the table is up-to-date, but we need to look into that.  Unfortunately that has a second order problem that we have to worry about: what happens when changes sink in that were done on the old table definition, particularly creates. I think creates are the only problem since updates should work fine and deletes should work fine. 

Another problem that we have to worry about is making sure that code that's running on an old version doesn't update the table definition back to the old schema (this is the only thing that I'm really not sure how to solve).

Lastly, I think this is a good opportunity and important time to enforce schema modification rules on tables. I've always intended that users should not delete columns out of a table and that includes rename or add required columns without simple default values (no code required to fill them in). If either of these cases are true, I think we should throw an error, but this is significant constraint imposed on the user code and it will be hard to surface it so we should think carefully about this. The code that already migrates tables to new schemes handles this with generic defaults, so it might be better to let that just work the way it is and have users deal with it when they aren't expecting that this allows them to have full control over the evolution of table scheme mask, but it could create some nasty data consistency problems
